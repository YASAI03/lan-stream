# Plan: lan-stream — ローカルウィンドウキャプチャ配信サーバー (Rust)

## TL;DR
特定のウィンドウ（EyeTracker等の隠れたウィンドウ含む）をGraphics Capture APIでキャプチャし、LAN内のブラウザからMJPEGストリームとして閲覧できるRust製サーバー。
設定はTOMLファイル + REST APIで管理。同時接続クライアントは1台のみ。1440pキャプチャ対応。

## アーキテクチャ概要

```
[Capture Thread]                          [Axum HTTP Server]
  Graphics Capture API → JPEG encode       GET /      → HTML viewer
       ↓                                   GET /raw   → MJPEG stream
  watch channel  ──── フレーム配信 ───→     GET/POST /api/config → 設定管理
                                            GET /api/windows → ウィンドウ一覧
```

## プロジェクト構造

```
lan-stream/
├── Cargo.toml
├── config.toml                # デフォルト設定
├── src/
│   ├── main.rs                # エントリーポイント
│   ├── config.rs              # 設定管理 (TOML読み書き + API)
│   ├── capture.rs             # ウィンドウキャプチャ (Graphics Capture API + D3D11)
│   ├── server.rs              # HTTP サーバー (axum) + ルーティング
│   └── stream.rs              # MJPEG ストリーミングロジック
```

## ステップ

### Phase 1: プロジェクト初期化
1. `cargo init` でプロジェクト作成
2. `Cargo.toml` に依存クレートを追加:
   - `windows` (Graphics Capture API, D3D11, Win32 GDI/Foundation/WindowsAndMessaging)
   - `axum` + `tokio` (HTTPサーバー)
   - `image` (JPEG エンコード。pure Rust、依存なし。1440pでボトルネックなら turbojpeg に差替え)
   - `serde` + `toml` (設定管理)
   - `tower` / `tower-http` (ミドルウェア)
   - `tokio-stream` / `async-stream` (MJPEG ストリーム生成)

### Phase 2: 設定管理 (`config.rs`, `config.toml`)
3. `config.toml` のスキーマ定義:
   ```toml
   [capture]
   window_title = ""   # キャプチャ対象ウィンドウのタイトル (部分一致)
   fps = 30
   quality = 80        # JPEG品質 1-100

   [server]
   host = "0.0.0.0"
   port = 8080
   ```
4. `config.rs`: TOML読み込み/書き込み、`Arc<RwLock<Config>>` で共有状態管理

### Phase 3: ウィンドウキャプチャ (`capture.rs`) — Graphics Capture API
5. `EnumWindows` でウィンドウ一覧を取得する関数
6. タイトル部分一致でターゲットウィンドウを検索
7. D3D11 デバイス作成 (`D3D11CreateDevice`)
8. `IGraphicsCaptureItemInterop::CreateForWindow(hwnd)` でキャプチャアイテム作成
9. `Direct3D11CaptureFramePool::CreateFreeThreaded` でフレームプール作成
10. `GraphicsCaptureSession` でキャプチャ開始
11. `FrameArrived` コールバックでフレーム受信 → GPU テクスチャをステージングテクスチャにコピー → CPU側にマップ
12. `image` クレートで BGRA → JPEG エンコード
13. 最新フレーム (JPEG bytes) を `tokio::sync::watch` で配信
14. ウィンドウが見つからない場合はリトライ (1秒間隔)

### Phase 4: MJPEG ストリーミング (`stream.rs`)
15. `multipart/x-mixed-replace; boundary=frame` レスポンス生成
16. `tokio::sync::watch` チャンネルでフレーム配信 (capture thread → HTTP handler)
17. 同時接続制限: `AtomicBool` フラグで1クライアントのみ許可

### Phase 5: HTTP サーバー (`server.rs`, `main.rs`)
18. axum ルーター構築:
    - `GET /` → HTMLページ (インライン。`<img src="/raw">` を含む)
    - `GET /raw` → MJPEG ストリーム (stream.rs)
    - `GET /api/config` → 現在の設定を JSON で返す
    - `POST /api/config` → 設定を更新し `config.toml` に永続化。キャプチャスレッドに反映
    - `GET /api/windows` → 表示可能なウィンドウ一覧 (タイトル + HWND)
19. `main.rs`: 設定読み込み → キャプチャスレッド起動 → axum サーバー起動

### Phase 6: HTMLビューワー
20. `/` で返す HTML:
    - `<img>` タグで `/raw` の MJPEG ストリームを表示
    - 接続状態の表示 (接続中 / 切断)
    - シンプルなスタイリング (ダークテーマ、中央配置)

## エンドポイント詳細

| メソッド | パス | 説明 | レスポンス |
|---------|------|------|-----------|
| GET | `/` | HTMLビューワー | `text/html` |
| GET | `/raw` | MJPEGストリーム | `multipart/x-mixed-replace` |
| GET | `/api/config` | 現在の設定取得 | `application/json` |
| POST | `/api/config` | 設定更新 (永続化) | `application/json` |
| GET | `/api/windows` | ウィンドウ一覧 | `application/json` |

## 検証

1. `cargo build` が成功すること
2. `cargo run` で起動し、別PCのブラウザから `http://<host>:8080/` でストリームが表示されること
3. `/raw` で MJPEG ストリームのみが表示されること
4. `/api/windows` でウィンドウ一覧が返ること
5. `/api/config` の GET/POST でキャプチャ対象を変更でき、再起動後も設定が維持されること
6. 2台目のクライアントが `/raw` にアクセスした際、適切に拒否されること

## 設計判断

- **Graphics Capture API を採用** (BitBlt ではなく):
  - 隠れたウィンドウ (EyeTracker) のキャプチャが必要
  - 1440p で GPU パスによる高パフォーマンスが有利
  - DirectX/OpenGL アプリも対応可能
  - Windows 10 1903+ 必須 (許容)
- **`image` クレート採用** (`turbojpeg` ではなく): pure Rust・依存なし。1440p 30fps でボトルネックになったら `turbojpeg` に差し替え
- **MJPEG 採用** (WebRTC/HLS ではなく): 最もシンプル。`<img>` タグで動作。ブラウザ側にJS不要 (映像のみの場合)
- **`watch` チャンネル** でフレーム配信: 最新フレームのみ保持。遅延クライアントがバッファを圧迫しない
- **スコープ外**: 音声キャプチャ、複数クライアント対応、認証、HTTPS
