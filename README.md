# lan-stream

Windows のウィンドウキャプチャを LAN 内に QOI delta over WebSocket で配信する軽量ストリーミングサーバー。

## モチベーション

- アイトラッカーを購入し、それを気軽に使えるようにしたいと考えた
- 会社用のPCに直接ソフトを入れるのは憚られるため、普段使っているメインのPCでキャプチャして、会社のPCからブラウザで見られるようにしたい
  - 作業しているモニターの裏側で動かすことによって、overlayすることも容易にできる

## 特徴

- **Windows Graphics Capture API** による高性能キャプチャ
  - 最小化・背面のウィンドウもキャプチャ可能
  - DirectX/OpenGL アプリ対応
  - GPU アクセラレーション（D3D11）
- **QOI delta encoding over WebSocket** でブラウザから直接視聴（プラグイン不要）
  - フレーム間 XOR 差分 → QOI 圧縮により、変化の少ない画面で帯域を大幅削減
  - 定期キーフレーム（~2秒間隔）で新規クライアント接続・エラー回復に対応
- **WASM デコーダー**（JS フォールバック付き）+ **WebGL レンダリング**（Canvas2D フォールバック付き）
- **透過 (アルファチャンネル) 完全対応** — EyeTracker 等の透過ウィンドウもそのまま配信
- 複数クライアント同時接続対応
- クライアントごとのビットレート計測
- Web UI で設定変更・パフォーマンスモニタリング
- 設定は `config.toml` に永続化

## 動作要件

- Windows 10 1903 以降（Graphics Capture API）
- Rust 2024 edition

## セットアップ

```bash
git clone <repo-url>
cd lan-stream
cargo run
```

起動後、ブラウザで `http://localhost:8080` にアクセス。

## 設定

`config.toml` で設定を管理する。Web UI（`/config`）からも変更可能。

```toml
[capture]
window_title = "メモ帳"      # キャプチャ対象（部分一致）
target_fps = 30              # 目標FPS (1-120)
capture_cursor = true        # マウスカーソルを含める

[server]
host = "0.0.0.0"             # バインドアドレス
port = 8080                  # ポート番号
```

| 項目 | 説明 | 範囲 | デフォルト |
|------|------|------|-----------|
| `window_title` | キャプチャするウィンドウのタイトル（部分一致） | 任意文字列 | `""` |
| `target_fps` | 目標フレームレート | 1 - 120 | 30 |
| `capture_cursor` | マウスカーソルをキャプチャに含めるか | `true` / `false` | `true` |
| `host` | サーバーのバインドアドレス | — | `0.0.0.0` |
| `port` | サーバーのポート番号 | 1 - 65535 | 8080 |

> **注意**: `host` / `port` の変更はサーバー再起動後に反映されます。

## ページ

| パス | 説明 |
|------|------|
| `/` | ストリームビューア — WebSocket 経由の QOI delta ストリームをリアルタイム表示 |
| `/raw` | ストリームビューア（代替ページ） |
| `/config` | 設定ページ — キャプチャ・サーバー設定を Web UI で変更 |
| `/debug` | デバッグダッシュボード — FPS、フレーム処理時間、ビットレート、サーバーログ |

全ページに共通ヘッダー（ナビゲーション＋キャプチャ状態＋ビットレート表示）を表示。

## API

| メソッド | パス | 説明 |
|----------|------|------|
| `GET` | `/ws` | WebSocket ストリーム（QOI delta バイナリフレーム＋ビットレート JSON） |
| `GET` | `/api/config` | 現在の設定を JSON で取得 |
| `POST` | `/api/config` | 設定を更新・永続化 |
| `GET` | `/api/windows` | キャプチャ可能なウィンドウ一覧 |
| `GET` | `/api/debug` | パフォーマンスメトリクス・サーバーログ・ビットレート |
| `GET` | `/api/health` | ヘルスチェック（キャプチャ状態、接続クライアント数、FPS、ビットレート） |
| `GET` | `/api/ping` | レイテンシ計測用 |

詳細な API 仕様は [docs/openapi.json](docs/openapi.json) を参照。

## アーキテクチャ

```
┌──────────────┐  broadcast::channel  ┌──────────────┐   WebSocket     ┌──────────┐
│  Capture     │ ── QOI delta ──────→ │  Axum Server │ ──────────────→ │ Browser  │
│  Thread      │   frames             │  (async)     │  (multiple)     │          │
│              │                      │              │ ← JSON API ──── │          │
│ D3D11 + GCA  │                      │ /ws          │                 │ /        │
│ + QOI delta  │                      │ /api/*       │                 │ /config  │
│   encode     │                      │ /config      │                 │ /debug   │
└──────────────┘                      └──────────────┘                 └──────────┘
```

- **キャプチャスレッド**: Graphics Capture API でウィンドウをキャプチャ → D3D11 ステージングテクスチャにコピー → CPU 読み出し → QOI delta エンコード → `broadcast::channel` で全フレーム配信
- **Delta Encoding**: 前フレームと現フレームを XOR → 差分を QOI 圧縮。画面の大部分が変化しない場合（デスクトップ操作、EyeTracker 等）、差分フレームは元の 1/5〜1/20 に圧縮される。定期的にキーフレーム（完全フレーム）を送信し、新規クライアント接続やデコードエラーからの回復を保証
- **WebSocket サーバー (Axum)**: `/ws` からフレームをバイナリメッセージで配信。毎秒テキストメッセージでクライアント個別ビットレートを通知。複数クライアント同時接続対応
- **フレーム配信**: `tokio::sync::broadcast` で全クライアントに全フレームを順序通り配信。delta encoding ではフレーム欠落が描画破綻に直結するため、遅延クライアントは次のキーフレームまでスキップして再同期する

## パフォーマンスメトリクス

デバッグダッシュボード（`/debug`）で以下を確認可能:

- **FPS**: 実測フレームレート
- **GPU Copy**: GPU テクスチャ → ステージングテクスチャのコピー時間
- **Map**: テクスチャのメモリマップ時間
- **Readback**: BGRA ピクセルデータの CPU 読み出し時間
- **Encode**: QOI エンコード時間
- **Total**: 1 フレームの合計処理時間
- **Bitrate**: WebSocket 全クライアント合計ビットレート

履歴チャート（約15分間）でトレンドも確認できる。

## 設計判断

### なぜ QOI + Delta Encoding か

| 要件 | 優先度 | 備考 |
|------|--------|------|
| **透過 (アルファチャンネル) 対応** | 必須 | EyeTracker 等の透過ウィンドウを扱うため |
| **FPS / レイテンシ維持** | 高 | エンコード速度がフレームレートを下げてはならない |
| **帯域削減** | 中 | LAN 前提だが、Wi-Fi や複数クライアントで有利 |

JPEG / H.264 はアルファチャンネル非対応のため脱落。AV1 / VP9 はソフトウェアエンコードが遅すぎ（1440p で 30fps 不可）。WebP lossy はアルファ対応だがエンコードが遅い（~30-60ms）。

QOI はロスレス・アルファ対応・高速エンコード（~8ms @ 1440p）だが、フレーム間圧縮がなく帯域が大きい。XOR delta encoding を上乗せすることで、QOI の速度とアルファ対応を維持しつつ、変化の少ない画面で帯域を 5-20x 削減する。

### 帯域の目安（1440p, 30fps）

| シナリオ | QOI (フレーム独立) | QOI + Delta |
|---------|-------------------|-------------|
| フルスクリーン (エディタ等) | 110-220 MB/s | 10-50 MB/s |
| EyeTracker (大半透過) | 1.5-9 MB/s | 0.06-0.9 MB/s |
